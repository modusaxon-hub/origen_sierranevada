BUILD-ID: 03032026-LOGO-002
HMR Activo — Cambios SVG en Tiempo Real
Completado
⚠ Riesgo sin este fix: La carpeta assets/ está fuera del project root de Vite. Sin configuración explícita, Vite no la observa — HMR nunca dispara para el logo.

Se implementó un plugin inline en vite.config.ts que registra la carpeta assets/ en el file watcher de Vite via server.watcher.add(). Adicionalmente se configuró server.fs.allow para permitir servir archivos desde fuera del project root. 

// vite.config.ts — Plugin watch-assets
{
  name: 'watch-assets',
  configureServer(server) {
    server.watcher.add(assetsPath); // assets/ registrada en chokidar
  },
}

Resultado: Modificar origen-logo-completo.svg → Vite detecta cambio → SVGR re-procesa → HMR actualiza el navegador en ~1-2 seg sin recargar.

BUILD-ID: 03032026-LOGO-003
Corrección de Proporciones — Header y Footer
Completado
⚠ Riesgo sin este fix: El logo en el footer (ratio 8.4:1) desbordaba su columna en 395px, causando que "SIERRA NEVADA" apareciera duplicado visualmente.

El SVG tiene ratio de aspecto 8.4:1 (590×70). Con h-20 w-auto en el footer (columna ~280px), el logo generaba un ancho de 675px desbordando la columna. Se corrigió usando w-full max-w-[220px] h-auto para controlar por ancho. En el Header, se eliminó el resize dinámico en scroll y se fijó a un tamaño estable.

Footer.tsx h-20 xl:h-24 w-auto → 675px ancho w-full max-w-[220px] h-auto proporcional
BUILD-ID: 03032026-LOGO-004
Puerto Fijo del Servidor — strictPort: true
Completado
⚠ Riesgo sin este fix: Sin strictPort, Vite migra automáticamente de puerto si el anterior está ocupado. El servidor había saltado 3001→3002→...→3179 en una sesión.

Se estableció port: 5000 con strictPort: true en vite.config.ts. El servidor ahora falla explícitamente si el puerto está ocupado, en lugar de migrar silenciosamente.

URL permanente: http://127.0.0.1:5000

Registro de Errores Resueltos

ERR-001 · server.watch.paths — TypeScript error 2769

Se intentó usar server.watch.paths para registrar la carpeta assets. Vite's server.watch acepta chokidar WatchOptions, que no incluye paths. Error: "Type '{ paths: string[] }' has no properties in common with type 'WatchOptions'"

Plugin inline con configureServer(server) { server.watcher.add(assetsPath) } — API correcta para registrar rutas adicionales en chokidar desde Vite.
ERR-002 · Logo duplicado en footer — "SIERRA NEVADA" ×2

El SVG con h-20 w-auto producía ~675px de ancho en una columna de ~280px. El texto del logo desbordaba hacia la columna de navegación adyacente, creando la apariencia de texto duplicado.

Cambio de estrategia: controlar con max-w en lugar de height.

Decisiones de Arquitectura (ADR)

ADR-001 · ¿Por qué vite-plugin-svgr en lugar de importar el SVG como URL?
Importar como URL (import logo from './logo.svg') entrega una string de ruta, no un componente. Para pasar className, aplicar brightness, drop-shadow y controlar el SVG vía CSS de Tailwind, se necesita el SVG inline como componente React.
Alternativa descartada: SVG estático via <img> — perdería control de estilos via CSS.
ADR-002 · ¿Por qué el alias @assets apunta fuera del project root?
El archivo origen-logo-completo.svg es el master corporativo del cliente, almacenado en assets/logo/svg/ fuera de web-page/pages/. Centralizar en assets/ previene duplicación cuando existan múltiples proyectos (ej. admin panel, PWA) que consuman el mismo logo.
Ruta: ../../assets desde web-page/pages/ → origen_sierranevada/assets/
ADR-003 · ¿Componentes separados para Header-Logo y Footer-Logo?
No. Un único <Logo className="..." /> con prop className es suficiente. Las diferencias de tamaño entre contextos se manejan con clases Tailwind (width vs height). Componentes separados generarían duplicación de lógica sin beneficio.
Regla: máxima cohesión, mínima duplicación. SVG escala perfectamente — no requiere variantes de componente.
