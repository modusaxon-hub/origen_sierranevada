Consolidaciones Recomendadas

Cada consolidacion reduce la superficie documental, elimina contradicciones entre archivos y crea una fuente de verdad unica por tema.

1. QA y Evaluacion de Calidad

REPORTE_PRUEBAS_FEEDBACK.md + REPORTE_PRUEBAS_FEEDBACK.html + REQUIREMENTS_EVALUATION.html → 04-OPERACIONES/QA_REPORTS.md
Razon: Los tres documentos hablan del mismo ciclo: verificacion de que el sistema cumple los requerimientos. El .md del reporte tiene el contenido mas actualizado (numero Nequi real, QR). El REQUIREMENTS_EVALUATION aporta la matriz de riesgos y el score de cumplimiento 88/100. El resultado fusionado debe tener: (1) Smoke Tests ejecutados, (2) Score de cumplimiento por categoria, (3) Gaps criticos abiertos, (4) Matriz de riesgos activos.

Estructura propuesta para QA_REPORTS.md: Fecha del ultimo test | Score global | Resultados por caso de prueba | Gaps criticos pendientes | Riesgos identificados | Historial de versiones del documento.

2. Requerimientos del Sistema (fuente de verdad unica)

TECHNICAL_SPECS.html + REQ_checkout_facturacion.html + REQ_selector_presentaciones.html → 02-ESPECIFICACIONES/REQUIREMENTS.md
Razon: Los tres describen QUE debe hacer el sistema. TECHNICAL_SPECS es el mas completo (RF-01 a RF-05 + casos de uso). Los dos REQ son especificaciones atomicas de modulos ya implementados que deben vivir como subsecciones dentro del documento mayor. La consolidacion debe respetar la estructura de casos de uso quirurgicos (UC-ADMIN-01, UC-ADMIN-02, UC-CLIENT-03) que son la joya del TECHNICAL_SPECS.

Estructura propuesta para REQUIREMENTS.md: Requisitos Funcionales (RF-01 a RF-0N) | Requisitos No Funcionales | Modulo Checkout y Facturacion (IVA incluido, campos DIAN) | Modulo Variantes de Producto | Casos de Uso detallados.

3. Guia de Entorno de Desarrollo

GITHUB_SETUP.html + REMOTE_ACCESS_GUIDE.html → 00-INICIALIZACION/AMBIENTE_LOCAL.md
Razon: Los dos documentos describen como preparar el ambiente donde se desarrolla. GITHUB_SETUP cubre el repositorio git (ya es historia pero util para nuevos colaboradores). REMOTE_ACCESS_GUIDE cubre el servidor Vite y el tunel Zrok — informacion operativa vigente. Un unico AMBIENTE_LOCAL.md que empiece con "clonar el repo" y termine con "el tunel esta activo en origen2025.share.zrok.io" es mas util que dos documentos separados.

Estructura propuesta para AMBIENTE_LOCAL.md: Prerequisitos | Clonar repositorio | Configurar .env (variables Supabase) | Iniciar servidor Vite | Activar tunel Zrok | Troubleshooting de red.

4. Guia de Diseno Responsivo

responsive_guidelines.html + RESPONSIVE_HIERARCHY.html → 03-GUIAS-DESARROLLO/RESPONSIVE_GUIDELINES.md
Razon: Ambos documentos definen el comportamiento responsivo del sistema. responsive_guidelines.html es el documento base (breakpoints precisos, touch targets 44px minimo, inputs iOS 16px minimo). RESPONSIVE_HIERARCHY.html anade la logica de jerarquia visual (orden de elementos en mobile, posicion de flechas de navegacion). La fusion es directa: responsive_guidelines como base + una seccion adicional de "Jerarquia Visual por Dispositivo" con el contenido de RESPONSIVE_HIERARCHY.

5. Historial de Cambios y Errores

CHANGELOG.html + ERROR_LOG.html → 05-CAMBIOS/CHANGELOG.md
Razon: El ERROR_LOG registra incidentes con fecha y solucion aplicada — que son exactamente el tipo de cambio que debe quedar en el CHANGELOG. Los tres errores resueltos (Zrok 404, IVA incorrecto, logo SVG) se pueden agregar como entradas historicas en el CHANGELOG.md bajo una seccion "Resolucion de Incidentes". Asi se tiene un unico registro temporal de todo lo que ha pasado en el proyecto.

6. Hoja de Ruta (extraida de MASTER_PLAN)

MASTER_PLAN.html (seccion Next Steps) → 05-CAMBIOS/ROADMAP.md
Razon: El MASTER_PLAN cumple dos roles distintos: documento de gobernanza historico (timeline de lo hecho) y planificacion futura (proximas fases). El timeline historico debe quedar en MASTER_PLAN.md. La seccion de "proximos pasos" (Wompi produccion, SEO dinamico) debe vivir en un ROADMAP.md separado que sea facil de actualizar en cada sesion sin tocar el documento de gobernanza.
